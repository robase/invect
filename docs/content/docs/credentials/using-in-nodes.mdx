---
title: Using Credentials in Nodes
description: How to reference stored credentials from node configuration.
---

Once you've created a credential (API key or OAuth2), you reference it from nodes that need authentication. The node receives the decrypted credential at execution time — you never hardcode secrets in node config.

## Attaching a credential

In the node config panel, the **Credential** dropdown lists all available credentials. Select one and it's attached to the node. When the flow runs:

1. The credential is loaded from the database
2. If it's an OAuth2 credential with an expired access token, it's refreshed first
3. The decrypted credential config is passed to the node's execution context
4. The node uses `context.credential.config.accessToken` (OAuth2) or `context.credential.config.apiKey` (API key) to authenticate

## In template expressions

You can reference the attached credential's values in template expressions using the `credential` namespace:

```
Authorization: Bearer {{ credential.accessToken }}
```

This works in any node parameter — URL, headers, body, prompt, etc. The credential is resolved before template evaluation.

## Credential types

Different node types expect different credential configurations:

**Model nodes** — expect an LLM credential with `provider` (`openai`, `anthropic`, `openrouter`) and `apiKey`. The Model node passes the key to the appropriate AI adapter automatically.

**HTTP Request nodes** — can use any credential type. You reference credential values explicitly in headers or body via template expressions. Common patterns:

```
# API key in header
Authorization: Bearer {{ credential.apiKey }}

# Basic auth
Authorization: Basic {{ btoa(credential.username + ":" + credential.password) }}
```

**Integration actions** (Gmail, Slack, GitHub, etc.) — expect an OAuth2 credential for the matching provider. The action accesses `context.credential.config.accessToken` internally.

## Multiple credentials in a flow

Different nodes in the same flow can use different credentials. A flow might use an OpenAI credential for a Model node, a Google OAuth2 credential for a Gmail node, and an API key for an HTTP request — each node references its own credential independently.

## Credential precedence

When a node executes, its credential is resolved in this order:

1. The credential explicitly attached to the node in its config
2. No fallback — if no credential is attached and the action requires one, execution fails with a clear error

There's no global credential or ambient auth. Every credential reference is explicit.
