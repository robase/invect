For services that use OAuth2 (Google, GitHub, Slack, Microsoft, etc.), Invect handles the complete flow: generating the authorization URL, exchanging codes for tokens, storing encrypted tokens, and refreshing them automatically.

## How it works

The OAuth2 flow in Invect follows the standard pattern:

1. You click "Connect" in the credentials UI and select a provider
2. You enter the **client ID** and **client secret** from your OAuth app
3. Invect generates an authorization URL and opens it in a popup
4. You authorize in the provider's UI (Google consent screen, etc.)
5. The provider redirects back with an authorization code
6. Invect exchanges the code for access and refresh tokens
7. Everything is encrypted and stored as a credential

After setup, nodes that use the credential get a valid access token automatically. If the token has expired, Invect refreshes it transparently before the node executes.

## Supported providers

35+ OAuth2 providers are built in, grouped by category:

**Google**: Gmail, Docs, Sheets, Drive, Calendar, Analytics
**Microsoft**: 365 (Mail, Calendar), Teams, OneDrive
**Developer tools**: GitHub, GitLab, Linear, Jira, Asana, Sentry
**Messaging**: Slack, Microsoft Teams
**CRM/Sales**: Salesforce, HubSpot
**Support**: Zendesk, Intercom, Freshdesk
**Commerce**: Shopify, Stripe, WooCommerce
**Social**: Twitter/X, LinkedIn, Facebook
**Storage**: Dropbox, OneDrive, Google Drive
**Other**: Notion, Trello, Grafana, Segment, Mixpanel, Resend, SendGrid

Each provider defines its authorization URL, token URL, required scopes, and whether it supports token refresh.

The full list is accessible via `GET /credentials/oauth2/providers`.

## Setting up your OAuth app

Before connecting a service, you need to create an OAuth application in that service's developer console and obtain a client ID and client secret. This is a one-time setup per provider.

For Google services, create a project in the [Google Cloud Console](https://console.cloud.google.com), enable the relevant APIs, and create OAuth 2.0 credentials. The callback URL should point to your Invect instance.

The specific steps vary by provider, but the pattern is always: create an app, get client ID/secret, configure the callback URL.

## Token refresh

Most providers issue both an access token (short-lived, usually 1 hour) and a refresh token (long-lived). When a node needs the credential, Invect checks whether the access token has expired:

- If not expired → use it directly
- If expired and refresh token exists → call the provider's token endpoint for a new access token
- If expired and no refresh token → the credential is invalid; the user needs to re-authorize

This happens transparently — nodes always receive a working token or a clear error.

Some providers (GitHub, Notion, Shopify) don't issue refresh tokens. For these, when the access token expires the user must re-authorize manually. The `GET /credentials/expiring` endpoint surfaces credentials approaching expiration.

## Frontend setup

The OAuth2 connection flow uses a popup window. The `@invect/ui` package includes the necessary callback handler. If you're embedding the editor, make sure the callback URL (`/oauth/callback`) is accessible in your frontend routing.

For custom frontends, the API calls are:

1. `POST /credentials/oauth2/start` → returns `{ authorizationUrl, state }`
2. Open `authorizationUrl` in a popup
3. After redirect, send code and state to `POST /credentials/oauth2/callback`
4. The callback creates the encrypted credential and returns it

## Adding a custom OAuth2 provider

If your service isn't in the built-in list, you can add a provider definition. See the [concepts: security](/docs/concepts/security) page for the credential system architecture.
