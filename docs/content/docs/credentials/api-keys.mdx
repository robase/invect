---
title: API Keys
description: Create, manage, and configure API key credentials.
---

Credentials in Invect store secrets — API keys, tokens, passwords — encrypted at rest with AES-256-GCM. You create them once and reference them from nodes that need authentication.

## Creating credentials

### In the UI

Go to **Credentials** in the sidebar, click **Add Credential**. The form asks for:

- **Name** — a human-readable label (e.g., "OpenAI Production")
- **Type** — `llm`, `http-api`, or `database`
- **Provider** — for LLM keys: `openai`, `anthropic`, or `openrouter`
- **Auth Type** — `API Key` for basic key-based auth
- **API Key** — the actual secret

### In config

For development, you can seed credentials on startup via `defaultCredentials` in your config:

```ts title="invect.config.ts"
export const invectConfig = defineConfig({
  // ...
  defaultCredentials: [
    {
      name: 'OpenAI',
      type: 'llm',
      provider: 'openai',
      authType: 'apiKey',
      config: { apiKey: process.env.OPENAI_API_KEY },
      isShared: true,
    },
    {
      name: 'Anthropic',
      type: 'llm',
      provider: 'anthropic',
      authType: 'apiKey',
      config: { apiKey: process.env.ANTHROPIC_API_KEY },
      isShared: true,
    },
  ],
});
```

These are created on startup if a credential with the same name doesn't already exist. Existing credentials are not overwritten.

### Via the API

```bash
curl -X POST http://localhost:3000/invect/credentials \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Stripe",
    "type": "http-api",
    "authType": "apiKey",
    "config": { "apiKey": "sk_live_..." },
    "isShared": true
  }'
```

## Encryption

All credential config values are encrypted before database storage. The encryption uses AES-256-GCM with a random IV per credential. The key comes from `INVECT_ENCRYPTION_KEY` (a 32-byte base64 string).

In development, if the env var isn't set, a deterministic default key is used. This is convenient but not secure — always set a real key in production. Generate one with:

```bash
npx invect-cli secret
```

See [Encryption](/docs/deployment/encryption) for production setup.

## Shared vs. private credentials

The `isShared` flag controls visibility:

- **Shared** (`isShared: true`) — visible to all users, usable in any flow
- **Private** (`isShared: false`) — visible only to the creator

When the auth plugin is active, credential listing respects user access. Private credentials are filtered out for non-owners.

## Editing and deleting

In the UI, click a credential to edit its name, metadata, or secret values. Deleting a credential removes it permanently — any nodes referencing it will fail on next execution.

Via the API: `PUT /credentials/:id` and `DELETE /credentials/:id`.

## Testing credentials

The UI provides a **Test** button that verifies the credential works. For LLM credentials, this makes a lightweight API call. For HTTP credentials, you can configure a test request.

Via the API: `POST /credentials/:id/test`.
