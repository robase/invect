## Input data construction

The `buildIncomingDataObject()` method creates the context for each node:

1. Find all upstream nodes (via edges)
2. For each upstream node, get its output from the execution store
3. Parse the output as JSON (if valid), otherwise use raw string
4. Key each output by the upstream node's `referenceId`

```ts
// Node Z has upstream nodes "Fetch User" (referenceId: fetch_user) and "Get Config" (referenceId: get_config)
const incomingData = {
  fetch_user: { id: 123, name: 'Alice' }, // JSON-parsed output
  get_config: 'production', // Raw string output
};
```

### Reference ID derivation

The `referenceId` is the node's label normalized to `snake_case`:

- "Fetch User" → `fetch_user`
- "API Response" → `api_response`

If two upstream nodes produce the same key, incrementing numbers are appended: `some_a`, `some_a1`.

## Template resolution

After building the input data, all template parameters are resolved:

1. Scan every config param for `{{ }}` patterns
2. For each match, evaluate the JavaScript expression via QuickJS WASM
3. The evaluation context includes all keys from the input data as local variables
4. Replace the template block with the result

Two evaluation modes:

| Mode            | When                         | Returns                              |
| --------------- | ---------------------------- | ------------------------------------ |
| Pure expression | Entire param is `{{ expr }}` | Raw JS value (object, array, number) |
| Mixed template  | Text + `{{ expr }}` blocks   | String (expressions stringified)     |

## Special node handling

### If/Else branching

When an If/Else node executes:

1. The condition expression is evaluated against the input data
2. The result determines which downstream branch is active
3. Nodes on the inactive branch are marked as `SKIPPED`
4. The If/Else node's output equals its input (passthrough)

### Agent execution

The Agent node has a different execution model — it runs an iterative loop:

1. Build the initial prompt from task prompt + system prompt + context
2. Send to LLM with available tool definitions
3. If LLM returns tool calls → execute tools → feed results back → repeat
4. If LLM returns text or stop tool → end loop
5. Final output = agent's last response

This loop is handled by `AgentNodeExecutor`, the only remaining legacy executor.

### Batch processing

When a Model node returns `state: "PENDING"`:

1. The node output includes a batch job ID
2. The flow run transitions to `PAUSED_FOR_BATCH`
3. The batch polling service periodically checks the provider
4. On completion, the flow resumes from the paused node

## State management

During execution, node outputs are stored in memory (per flow run). After the run completes, the full execution trace (all node inputs, outputs, errors, timing) is persisted to the database.

Each node execution record includes:

| Field                       | Description                             |
| --------------------------- | --------------------------------------- |
| `nodeId`                    | The node's ID within the flow           |
| `status`                    | `COMPLETED`, `FAILED`, `SKIPPED`        |
| `input`                     | The incoming data object                |
| `resolvedParams`            | Config params after template resolution |
| `output`                    | The node's result                       |
| `error`                     | Error message (if failed)               |
| `startedAt` / `completedAt` | Timing                                  |
