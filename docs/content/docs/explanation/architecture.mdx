Invect is a library, not a platform. You install npm packages, mount a router, and your existing Node.js app gains workflow orchestration. This page explains how the pieces fit together.

## The core-adapter pattern

All business logic lives in `@invect/core`. Framework packages (`@invect/express`, `@invect/nestjs`, `@invect/nextjs`) are thin adapters that translate HTTP requests into core method calls.

```
┌─────────────────────────────────────┐
│          @invect/core               │
│                                     │
│   Invect class                      │
│   ├── createFlow()                  │
│   ├── startFlowRun()                │
│   ├── listFlows()                   │
│   ├── createCredential()            │
│   └── ... 40+ methods               │
│                                     │
│   Services  │  Actions  │  Database │
└──────────┬──────┬──────┬────────────┘
           │      │      │
     ┌─────┘      │      └─────┐
     │            │            │
  Express      NestJS       Next.js
  Router       Module       Handler
  (~300 LOC)   (~400 LOC)   (~300 LOC)
```

Each adapter does the same thing:

1. Creates an `Invect` instance with your config.
2. Calls `initialize()` (sets up database, registers actions, starts schedulers).
3. Maps HTTP routes to `core.<method>()` calls.
4. Returns results as JSON.

This means adding a new framework adapter is straightforward — import `Invect`, wire routes, delegate.

## Service layer

Inside `@invect/core`, business logic is organised into services. No controller or adapter accesses the database directly — everything goes through a service.

```
FlowOrchestrationService          ← Orchestrates complete flow execution
├── FlowRunCoordinator            ← Manages the lifecycle of a single flow run
├── NodeExecutionCoordinator      ← Executes individual nodes, resolves templates
├── FlowsService                  ← Flow CRUD
├── FlowVersionsService           ← Version management
├── FlowRunsService               ← Execution records
├── NodeExecutionService          ← Node-level traces
├── BatchJobsService              ← Batch processing integration
└── CredentialsService            ← Encrypted credential storage

GraphService                      ← Topological sorting, dependency analysis
TemplateService                   ← JavaScript expression evaluation (QuickJS)
ReactFlowRendererService          ← Prepares flow data for the visual editor
```

Services are instantiated by `ServiceFactory`, which handles dependency injection. Each service receives a `DatabaseService` and any other services it depends on.

## Database abstraction

Invect supports SQLite, PostgreSQL, and MySQL through Drizzle ORM. The `DatabaseService` wraps a Drizzle connection and provides typed access to all tables.

Models (in `pkg/core/src/models/`) handle CRUD operations. They receive the Drizzle connection and database type, and use Drizzle's query builder — no raw SQL.

```
Config → DatabaseService → Drizzle connection → Models → Services
```

Schema files are generated by the CLI (`invect-cli generate`) from an abstract schema definition. This abstract format lets plugins declare tables without knowing which database the user runs.

## Action registry

Every node type (except the legacy Agent node) is defined as an **action** — a self-contained unit that includes its configuration schema, UI field definitions, and execution function.

Actions are registered in the `ActionRegistry` during `Invect.initialize()`. The registry serves two purposes:

1. **Flow nodes** — the visual editor reads action definitions to populate its node palette.
2. **Agent tools** — each action is automatically converted into an agent tool definition.

This dual-purpose design means adding a new integration (e.g., a Slack action) simultaneously creates a flow node and an agent tool, with no extra work.

## Plugin manager

Plugins are registered via the `plugins` config array. The `PluginManager` handles:

- Registering plugin actions in the action registry.
- Calling plugin `init()` functions in declaration order.
- Executing lifecycle hooks (beforeFlowRun, onRequest, etc.) at the right moments.
- Routing requests to plugin endpoints.
- Calling `shutdown()` in reverse order on graceful exit.

Plugins can't modify core behaviour directly — they intercept and extend through well-defined hook points.

## Frontend

The visual editor (`@invect/ui`) is a React application that communicates with the backend exclusively through the REST API. It is completely decoupled from the server — you can host it separately, embed it in an existing React app, or skip it entirely and use only the API.

The `Invect` component provides the full app shell (sidebar, router, editor). The `InvectShell` component provides only the CSS scope (theme tokens) for cases where you need Invect styling without the full app — for example, rendering a sign-in page before the editor loads.

## Key design decisions

- **Library, not platform**: You own the server, the database, and the deployment. Invect is a dependency.
- **Service layer**: All logic goes through services. No shortcuts from routes to database.
- **Thin adapters**: Framework packages are intentionally minimal. Business logic never lives in an adapter.
- **Dual-purpose actions**: One definition serves both the visual editor and the agent tool system.
- **Generated schemas**: The CLI generates database schemas from an abstract format, letting plugins contribute tables without dialect-specific code.
