Template expressions let you reference upstream node outputs and write JavaScript directly in node config fields. They're evaluated in a sandboxed **QuickJS WASM** environment — no access to Node.js APIs, the filesystem, or the network.

## Syntax

Wrap any JavaScript expression in double curly braces:

```
{{ expression }}
```

The expression is evaluated with all upstream node outputs available as local variables, keyed by each node's `referenceId`.

## Examples

### Simple property access

```
{{ input.topic }}
{{ api_result.data.users }}
```

### Full JavaScript

```
{{ input.items.filter(item => item.active).length }}
{{ JSON.stringify(fetch_user, null, 2) }}
{{ Math.round(score * 100) / 100 }}
```

### String interpolation

Mix static text with expressions:

```
Hello {{ input.name }}, you have {{ orders.length }} orders.
```

### Conditional values

```
{{ input.priority === 'high' ? 'URGENT: ' : '' }}{{ input.message }}
```

## How it works

### Pure vs mixed expressions

There are two evaluation modes:

| Mode      | Pattern                                | Returns                                               |
| --------- | -------------------------------------- | ----------------------------------------------------- |
| **Pure**  | Entire value is a single `{{ expr }}`  | Raw JavaScript value (object, array, number, boolean) |
| **Mixed** | Text with embedded `{{ expr }}` blocks | Always a string, each expression stringified          |

**Pure expression** — returns the raw value:

```
{{ input.users }}
// Returns: [{ name: "Alice" }, { name: "Bob" }] (array)
```

**Mixed expression** — returns a string:

```
Found {{ input.users.length }} users
// Returns: "Found 2 users" (string)
```

This matters when passing structured data between nodes. If you need to pass an object or array, use a pure expression (the entire field value is `{{ expr }}`).

### Context variables

Every upstream node's output is available as a variable named after the node's `referenceId`:

```json
{
  "input": { "topic": "AI", "count": 5 },
  "fetch_data": { "results": [...] },
  "transform": "processed string"
}
```

The `referenceId` is derived from the node's label, normalized to `snake_case`. If two upstream nodes produce the same key, Invect appends incrementing numbers (`some_a`, `some_a1`).

A special `$input` variable is always available as an alias to the entire incoming data object.

### Error handling

In **mixed** templates, if an expression throws an error, it produces an empty string (the rest of the template still renders). In **pure** templates, errors propagate and the node execution fails.

## Common patterns

### Accessing nested data

```
{{ api_response.data.items[0].name }}
{{ config.settings?.theme ?? 'default' }}
```

### Transforming arrays

```
{{ users.map(u => u.email).join(', ') }}
{{ orders.filter(o => o.status === 'pending') }}
```

### Building prompts

```
Summarize the following {{ input.items.length }} items:

{{ input.items.map((item, i) => `${i + 1}. ${item.title}`).join('\n') }}
```

### Passing structured data

When a node needs a JSON object as input, use a pure expression:

```
{{ { name: input.name, email: input.email, role: 'user' } }}
```

<Callout type="warn">
  Template expressions run in a WASM sandbox with no access to `fetch`, `fs`, `process`, or other
  Node.js APIs. They're for data transformation only — use HTTP Request or custom action nodes for
  I/O.
</Callout>
