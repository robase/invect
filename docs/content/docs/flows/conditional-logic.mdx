---
title: Conditional Logic
description: Branch flows with if/else and switch nodes.
---

Not every flow is a straight pipeline. Conditional nodes let you branch execution based on data values, sending downstream work down different paths.

## If/Else nodes

The **If/Else** node evaluates a condition against its incoming data and routes execution to one of two branches: **true** or **false**.

Add an If/Else node from the Core provider. In its config panel, set the **condition** — a template expression that should evaluate to a truthy or falsy value:

```
{{ api_response.status === 200 }}
{{ user.role === "admin" }}
{{ items.length > 0 }}
```

The node has two output handles. Connect downstream nodes to either the `true` or `false` handle. When the flow runs, only nodes on the matching branch execute — the other branch is skipped entirely.

**Key behaviour:** If/Else nodes are **passthrough**. Their output equals their input. Downstream nodes on the active branch receive the same data that flowed into the conditional — the condition itself doesn't produce new data.

## Switch nodes

The **Switch** node extends branching to multiple paths. Instead of true/false, it evaluates a value and matches it against named cases.

Configure the **value** expression (what to evaluate) and define **cases** (each with a label and match value). There's also a default case for unmatched values.

```
Value: {{ order.status }}
Cases:
  - "pending" → nodes for pending orders
  - "shipped" → nodes for shipped orders
  - "delivered" → nodes for delivered orders
  - default → error handling nodes
```

Like if/else, switch nodes are passthrough — they route execution without transforming data.

## Nesting conditions

You can chain conditional nodes. An if/else node's `true` branch can feed into another if/else, creating decision trees. Each condition evaluates against the data flowing through its specific path.

## Impact on downstream nodes

When a branch is inactive, all nodes reachable only through that branch are marked as `SKIPPED` in the execution trace. They don't execute and they don't produce output. This works recursively — a skipped branch means everything downstream of it is also skipped.

If a node is reachable through multiple paths (e.g., both branches eventually merge), it executes as long as at least one of its upstream paths is active.
