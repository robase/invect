Understanding how Invect executes flows helps you build reliable workflows and debug issues.

## Execution order

Invect uses **topological sorting** to determine the order nodes execute. A node only runs after all its upstream dependencies have completed. Nodes at the same level with no dependencies between them could theoretically run in parallel (currently sequential).

```
Input ──→ Fetch Data ──→ Transform ──→ Output
              │
              └──→ Validate ──→ Log
```

In this flow, execution order is:
1. `Input`
2. `Fetch Data` (depends on Input)
3. `Transform` and `Validate` (both depend on Fetch Data)
4. `Output` (depends on Transform) and `Log` (depends on Validate)

## Data flow

### Building input data

Before executing a node, Invect constructs its **incoming data object** from all upstream node outputs:

```json
{
  "fetch_data": { "users": [{ "name": "Alice" }] },
  "validate": { "valid": true }
}
```

The keys are each upstream node's `referenceId` (derived from the node label, normalized to `snake_case`). The values are the parsed output of each upstream node.

### Template resolution

After building the input data, Invect resolves all template expressions in the node's config params. Each `{{ expression }}` is evaluated with the input data as the JavaScript context.

```
Config: "Found {{ fetch_data.users.length }} users"
Result: "Found 1 users"
```

See [Template Expressions](/docs/flows/templates) for full details.

### Output storage

Each node's output is stored and made available to downstream nodes. Outputs are JSON-parsed if valid JSON, otherwise stored as raw strings.

## Flow run lifecycle

```
┌─────────────┐
│  PENDING     │  Flow run created
└──────┬──────┘
       ▼
┌─────────────┐
│  RUNNING     │  Nodes executing in order
└──────┬──────┘
       ▼
┌─────────────────────────────────────┐
│  SUCCESS / FAILED / PAUSED_FOR_BATCH │
└─────────────────────────────────────┘
```

| Status | Description |
|--------|-------------|
| `PENDING` | Flow run created, not yet started |
| `RUNNING` | Nodes are actively executing |
| `SUCCESS` | All nodes completed successfully |
| `FAILED` | A node failed and the flow stopped |
| `PAUSED_FOR_BATCH` | A node returned a batch job — flow pauses until the batch completes |
| `CANCELLED` | Flow run was cancelled |

### Node execution states

Each individual node goes through its own lifecycle:

| Status | Description |
|--------|-------------|
| `PENDING` | Waiting to execute |
| `RUNNING` | Currently executing |
| `COMPLETED` | Finished successfully |
| `FAILED` | Execution failed |
| `SKIPPED` | Skipped (inactive branch of an if/else) |

## Execution tracing

When `execution.enableTracing` is enabled (default: `true`), Invect records a `NodeExecution` trace for every node run. Each trace includes:

- Input data (the incoming data object)
- Resolved parameters (after template processing)
- Output data
- Error details (if failed)
- Timing information

Access traces via the UI (click a node after running) or the API:

```bash
curl http://localhost:3000/invect/flow-runs/{runId}
```

## Error handling

When a node fails, the flow run is marked as `FAILED` and execution stops. Downstream nodes are not executed.

The error is captured in the node execution trace with the full error message and stack trace.

## Timeouts

Configure timeouts in [execution config](/docs/reference/configuration):

| Option | Default | Description |
|--------|---------|-------------|
| `defaultTimeout` | 60,000ms | Per-node execution timeout |
| `flowTimeoutMs` | 600,000ms (10 min) | Total flow execution timeout |
| `heartbeatIntervalMs` | 30,000ms | Heartbeat interval for long-running flows |
| `staleRunCheckIntervalMs` | 60,000ms | How often to check for stale runs |

## Concurrency

The `maxConcurrentExecutions` config (default: `10`) limits how many flow runs can execute simultaneously. Additional runs are queued.
