Batch processing lets you submit large workloads to AI provider batch APIs (OpenAI, Anthropic) at reduced cost. When a node returns a batch job, the flow **pauses** and automatically **resumes** when the batch completes.

## How it works

1. A Model node submits work to the provider's batch API
2. The provider returns a batch job ID with status `PENDING`
3. The flow run transitions to `PAUSED_FOR_BATCH`
4. Invect's batch polling service checks the provider periodically
5. When the batch completes, the flow resumes from where it paused

```
Node executes → Batch submitted → Flow pauses → Polling → Batch completes → Flow resumes
```

## Batch polling

Batch polling starts automatically when you create an Invect router or initialize the core. It periodically checks the status of pending batch jobs and resumes paused flows.

The polling service is managed internally — no additional configuration is required.

## Flow run status

During batch processing, the flow run has a special status:

| Status | Meaning |
|--------|---------|
| `PAUSED_FOR_BATCH` | Flow is waiting for a batch job to complete |
| `RUNNING` | Flow resumed after batch completion |
| `SUCCESS` | All nodes (including post-batch) completed |

## API

```bash
# List batch jobs
curl http://localhost:3000/invect/batch-jobs

# Get a specific batch job
curl http://localhost:3000/invect/batch-jobs/{batchJobId}
```

## When to use batch processing

Batch APIs are typically 50% cheaper than real-time API calls but take longer (minutes to hours). Use them when:

- Processing large datasets that don't need real-time results
- Running overnight analytics or report generation
- Cost optimization is more important than latency
