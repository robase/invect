## Core + adapter pattern

All business logic lives in `@invect/core`. Framework packages are thin adapters that wrap the core. This means the same logic runs identically across Express, NestJS, and Next.js.

## Service layer

Services handle all business logic. They're created by the `ServiceFactory` during initialization and injected into the `Invect` class.

| Service                    | Responsibility                                          |
| -------------------------- | ------------------------------------------------------- |
| `FlowOrchestrationService` | Orchestrates complete flow execution                    |
| `FlowRunCoordinator`       | Coordinates full flow run lifecycle                     |
| `NodeExecutionCoordinator` | Handles individual node execution + template resolution |
| `FlowsService`             | Flow CRUD operations                                    |
| `FlowVersionsService`      | Flow version management                                 |
| `FlowRunsService`          | Flow execution records                                  |
| `NodeExecutionService`     | Node-level execution traces                             |
| `BatchJobsService`         | Batch processing integration                            |
| `CredentialsService`       | Secure credential storage + OAuth2 refresh              |
| `TemplateService`          | Template rendering via QuickJS WASM sandbox             |
| `GraphService`             | Topological sorting, dependency analysis                |
| `ReactFlowRendererService` | Flow visualization data for frontend                    |

**Services never bypass each other.** Controllers/routes delegate to the `Invect` class, which delegates to services, which delegate to models for database access.

## Database layer

Invect uses **Drizzle ORM** for database access, supporting three dialects:

| Dialect    | Driver Options                                    |
| ---------- | ------------------------------------------------- |
| SQLite     | `better-sqlite3`, `libsql`                        |
| PostgreSQL | `postgres` (postgres.js), `pg`, `neon-serverless` |
| MySQL      | `mysql2`                                          |

The database layer is abstracted through **Models** — each model handles CRUD for a specific table:

```ts
class FlowsModel {
  constructor(
    private db: DatabaseConnection,
    private dbType: DatabaseType,
  ) {}

  async create(data: NewFlow): Promise<Flow> {
    return this.db.insert(flows).values(data).returning();
  }
}
```

**Never write raw SQL** — use Drizzle ORM methods through models.

## Initialization sequence

When you call `createInvectRouter()`, `InvectModule.forRoot()`, or `createInvectHandler()`:

1. **Action registry created** — all built-in actions registered (`defineAction()` definitions)
2. **Plugin initialization** — for each plugin: register `plugin.actions`, call `plugin.init(context)`
3. **Node registry initialized** — registers `AgentNodeExecutor` (only node executor)
4. **Actions registered as tools** — every action converted to an `AgentToolDefinition`
5. **Services initialized** — `ServiceFactory` built, `DatabaseService.initialize()` runs
6. **Table verification** — core tables checked, then plugin tables checked
7. **Batch polling started** — periodic check for completed batch jobs
8. **Cron scheduler started** — if enabled, starts scheduling cron-triggered flows

## Plugin integration

Plugins are registered during initialization and can contribute:

- **Actions** — new node types/agent tools
- **Database schema** — new tables or columns on existing tables
- **API endpoints** — custom routes
- **Hooks** — lifecycle hooks that intercept execution
- **Frontend components** — sidebar items, routes, panel tabs

Plugins are initialized in order and shut down in reverse order. See [Plugin System](/docs/plugins/overview) for details.
