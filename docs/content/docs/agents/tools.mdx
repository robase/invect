Agent tools are what give agents their power — they're the functions the LLM can call during the iterative loop. Invect generates its tool list automatically from the action registry, plus a few standalone utilities.

## Where tools come from

### Action-based tools

Every action registered with `defineAction()` is automatically converted to an agent tool during startup. The conversion maps:

- Action `id` → tool `id`
- Action `description` → tool description (what the LLM sees)
- Action `params.schema` (Zod) → tool input schema (JSON Schema)
- Action `execute` → tool executor

This means all 190+ built-in actions (Gmail, Slack, GitHub, HTTP, JQ, etc.) are available as tools without extra configuration.

### Standalone tools

Two utility tools are registered separately (not backed by a flow node):

| Tool         | Description                       |
| ------------ | --------------------------------- |
| `math_eval`  | Evaluate mathematical expressions |
| `json_logic` | Apply JsonLogic rules to data     |

These are for agent use only — they don't appear in the node palette.

## Selecting tools per agent

In the agent node's config panel, the **Enabled Tools** selector lets you pick which tools the agent can access. The full list is available from the tool selector modal, grouped by provider.

Be selective — giving an agent 100 tools makes the LLM's `description` prompts very long and the model less likely to choose wisely. Enable only the tools relevant to the agent's task.

## Per-instance tool configuration

Each tool instance on an agent can be customised:

- **Custom name** — override the default tool name (useful for clarity when the same tool appears multiple times)
- **Custom description** — override what the LLM sees about the tool
- **Fixed parameters** — lock certain parameters so the LLM can't change them (e.g., always send to a specific Slack channel)

This lets you create specialised tool instances without defining new actions.

## Custom tools

You have two options for adding tools:

### Option A: Create an action (recommended)

Define a new action with `defineAction()`. It automatically becomes both a flow node and an agent tool. See [Custom actions](/docs/nodes/custom-actions).

### Option B: Standalone tool

For tools that don't make sense as flow nodes, register them directly:

```ts
import type { AgentToolDefinition, AgentToolExecutor } from '@invect/core';

export const myToolDefinition: AgentToolDefinition = {
  id: 'my_custom_tool',
  name: 'My Custom Tool',
  description: 'What this tool does — the LLM reads this to decide when to use it',
  inputSchema: {
    type: 'object',
    properties: {
      query: { type: 'string', description: 'The search query' },
    },
    required: ['query'],
  },
  category: 'utility',
  enabledByDefault: false,
};

export const myToolExecutor: AgentToolExecutor = async (input, context) => {
  const result = await doSomething(input.query);
  return { success: true, output: result };
};
```

Register standalone tools in `pkg/core/src/services/agent-tools/builtin/index.ts`.

## How tools are exposed to LLMs

When the agent sends a request to the LLM (OpenAI or Anthropic), Invect converts the enabled tools into the provider's format:

- **OpenAI**: Function calling format (`tools` array with `function` type)
- **Anthropic**: Tool use format (`tools` array with `input_schema`)

The conversion handles schema differences between providers automatically. The LLM sees tool names, descriptions, and JSON Schema inputs — it never sees the implementation.

## Tool credentials

Tools that require credentials (OAuth2 tokens, API keys) use a credential attached to the agent node or specified in the tool instance configuration. The credential is automatically decrypted and refreshed before the tool executes.
