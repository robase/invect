Agents use **tools** to interact with the world. Every action registered in Invect is automatically available as an agent tool, plus there are standalone utility tools.

## Tool sources

### Action-based tools (primary)

Every action defined with `defineAction()` is automatically registered as both a flow node **and** an agent tool during initialization. This means all 190+ integration actions are available to agents:

- Gmail actions → `gmail.send_message`, `gmail.list_messages`
- Slack actions → `slack.send_message`, `slack.list_channels`
- GitHub actions → `github.create_issue`, `github.list_repos`
- HTTP Request → `http.request`
- JQ → `core.jq`
- And every other registered action

### Standalone tools

Utility tools that only exist as agent tools (no corresponding flow node):

| Tool ID | Description |
|---------|-------------|
| `math_eval` | Evaluate mathematical expressions safely |
| `json_logic` | Apply JSON Logic rules to data |

## Enabling tools

In the Agent node config, you select which tools the agent can use. The tool selector shows all available tools organized by provider.

Each agent instance can have a different set of enabled tools. This lets you create focused agents — one for email tasks (Gmail tools only), another for data processing (JQ + HTTP tools), etc.

## Tool execution context

When a tool executes in an agent context, it receives:

| Context | Description |
|---------|-------------|
| `credential` | OAuth2 tokens or API keys (auto-refreshed) |
| `incomingData` | The agent's current execution data |
| `flowInputs` | Original flow inputs |
| `logger` | Structured logger |
| `functions` | Service functions (getCredential, submitPrompt, etc.) |

Tools that require credentials (OAuth2 actions) are automatically provided with the credential configured on the agent node.

## Per-instance configuration

Each tool instance on an agent can be customized:

| Setting | Description |
|---------|-------------|
| Custom name | Override the tool name shown to the LLM |
| Custom description | Provide more specific instructions for when to use it |
| Custom parameters | Pre-fill or constrain tool parameters |

This is useful for making generic tools more specific. For example, an `http.request` tool could be customized with a fixed base URL and description like "Fetch user data from the internal API".

## API

```bash
# List all available agent tools
curl http://localhost:3000/invect/agent/tools
```

Returns an array of tool definitions with `id`, `name`, `description`, `inputSchema`, and `category`.

## Adding custom tools

### Option A: Create a new action (recommended)

The action automatically becomes both a flow node and an agent tool:

```ts title="pkg/core/src/actions/my-provider/my-action.ts"
import { defineAction } from '../define-action';
import { z } from 'zod/v4';

export const myAction = defineAction({
  id: 'my_provider.my_action',
  name: 'My Action',
  description: 'What this does — the LLM reads this to decide when to use it',
  provider: MY_PROVIDER,
  params: {
    schema: z.object({ query: z.string() }),
    fields: [{ name: 'query', label: 'Query', type: 'text', required: true }],
  },
  async execute(params, context) {
    const result = await doSomething(params.query);
    return { success: true, output: result };
  },
});
```

### Option B: Standalone tool (utility only)

For tools that don't make sense as flow nodes:

```ts title="pkg/core/src/services/agent-tools/builtin/my-tool.ts"
export const myToolDefinition: AgentToolDefinition = {
  id: 'my_tool',
  name: 'My Tool',
  description: 'Describe what this tool does for the LLM',
  inputSchema: { /* JSON Schema */ },
  category: 'utility',
  enabledByDefault: true,
};

export const myToolExecutor: AgentToolExecutor = async (input, context) => {
  return { result: 'done' };
};
```
