## Node parameters

| Param                 | Type       | Default           | Description                                                                           |
| --------------------- | ---------- | ----------------- | ------------------------------------------------------------------------------------- |
| `credentialId`        | credential | —                 | OpenAI, Anthropic, or OpenRouter API key                                              |
| `model`               | string     | —                 | Model name (e.g., `gpt-4o-mini`, `claude-sonnet-4-20250514`)                          |
| `taskPrompt`          | string     | —                 | Main task/goal for the agent (supports [template expressions](/docs/flows/templates)) |
| `systemPrompt`        | string     | `""`              | Optional system instructions                                                          |
| `enabledTools`        | string[]   | `[]`              | Array of tool IDs to enable                                                           |
| `maxIterations`       | number     | `10`              | Maximum number of prompt→tool→response loops (1–50)                                   |
| `stopCondition`       | string     | `"explicit_stop"` | When to stop: `explicit_stop`, `tool_result`, `max_iterations`                        |
| `enableParallelTools` | boolean    | `false`           | Allow the LLM to call multiple tools per response                                     |

## Provider-specific behavior

### OpenAI

- Uses the **function calling** format
- Supports parallel tool calls natively
- Token streaming for real-time output
- Models: `gpt-4o`, `gpt-4o-mini`, `gpt-4-turbo`, `o1`, `o3-mini`

### Anthropic

- Uses the **tool_use** content block format
- Streaming via SSE
- Models: `claude-sonnet-4-20250514`, `claude-sonnet-4-20250514`, `claude-3-haiku`

### OpenRouter

- Extends the OpenAI adapter format
- Routes to multiple underlying providers
- Access models from DeepSeek, Meta, Mistral, Google, and more
- Uses OpenAI-compatible tool calling

## Stop conditions explained

### `explicit_stop`

The agent must call a special stop tool to signal it's finished. This is the best choice for open-ended tasks where the agent needs to gather information, process it, and produce a final answer.

The stop tool accepts a `result` parameter — the agent's final output.

### `tool_result`

The agent stops immediately after executing its first tool call. Useful for simple patterns like "look up this data and return it" where one tool call is sufficient.

### `max_iterations`

The agent runs until it hits the iteration limit. This is always active as a safety net — even with other stop conditions, the agent will stop at `maxIterations`.

## Tips for effective agents

**Write clear task prompts.** The more specific your task prompt, the fewer iterations the agent needs. Include the expected output format.

**Limit tools.** Only enable tools the agent needs. Fewer tools = less confusion for the LLM and faster responses.

**Use system prompts for constraints.** Put behavioral rules in the system prompt: "Always verify data before returning", "Format output as JSON", etc.

**Set appropriate iteration limits.** Simple tasks need 2–5 iterations. Complex research tasks may need 10–20. Setting the limit too high wastes API calls on stuck agents.

**Use template expressions for context.** Pass upstream data into the task prompt so the agent has the context it needs without making extra tool calls.
